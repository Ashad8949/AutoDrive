What We Built

An AI car-shopping assistant for AutoDrive — a live Indian pre-owned car marketplace — that answers real inventory questions, compares cars in ₹, and books test drives. Three Azure student accounts collaborating in one live system.

  • Live inventory sync from friend's PostgreSQL via REST API
  • Streaming LLM responses (Server-Sent Events, first token <1s)
  • Voice input (Groq Whisper STT) + browser audio output (TTS)
  • Floating chat widget integrated in friend's Next.js frontend
  • Deep-link: bot names a car → one click → car details page
  • Test drive booking widget embedded inline in chat
  • Chatbot answers in ₹ — Maruti, Hyundai, Tata, etc.

AI / RAG
05

What is RAG?

Retrieval Augmented Generation — Instead of memorising facts (like fine-tuning), the LLM fetches relevant data at query time and uses it as context. Open-book exam vs closed-book.

👤 User Query
SUVs under ₹20L
→
🔍 TF-IDF
Score all 12 cars
→
📄 Context
Top 3-5 car docs
→
🤖 LLaMA 3.3
Generates response
→
✅ Answer
In ₹, real data

Step 1 — Retrieve

Query "SUVs under ₹20L" is scored against all 12 cars using TF-IDF (Term Frequency–Inverse Document Frequency). Top matches: Creta, Seltos, Hector, Nexon EV.

Step 2 — Augment

Matched car details (price in ₹, mileage, features, location, rating) are injected into the LLM's system prompt. The model now "knows" the inventory.

Step 3 — Generate

LLaMA 3.3-70B generates a grounded response using only the provided context. Correct ₹ prices, actual features, real cities. Zero hallucination on inventory data.

Why TF-IDF works here

  • Car inventory = structured keyword-rich text (make, model, fuel, price)
  • TF-IDF excels at exact keyword matching
  • Zero API cost — runs locally in-process
  • Sub-millisecond retrieval for ≤30 cars

Upgrade path (100+ cars)

  • Switch to Azure AI Search with vector embeddings
  • One config change in config.py
  • Architecture already supports it — retriever is pluggable
  • text-embedding-3-large deployment already exists in Azure

Containerisation — Docker

Multi-stage build: build tools stay in the builder stage, only the runtime lands in production. Result: small, secure, reproducible image.

# Stage 1 — Builder
FROM python:3.12-slim AS builder
WORKDIR /app
COPY requirements.txt .
RUN pip install --prefix=/install -r requirements.txt

# Stage 2 — Production (minimal)
FROM python:3.12-slim
WORKDIR /app
COPY --from=builder /install /usr/local
COPY src/ ./src/
ENV PYTHONPATH=/app
CMD ["uvicorn", "src.main:app",
     "--host", "0.0.0.0",
     "--port", "8002", "--workers", "2"]

Free for public packages. Image: ghcr.io/ashad8949/autodrive-chatbot. Replaced Azure Container Registry (~$5/month).

Test Gate

All 5 unit tests must pass. Tests use mocked engine — no real API calls. If any fail, pipeline stops. Code never ships broken.

Build Gate

Staging Gate

Auto-Rollback

After production slot swap, re-checks /health. If it fails → automatically swaps back. Zero manual intervention. Zero downtime.

Blue-Green Deployment — Zero Downtime

Azure App Service deployment slots run two versions simultaneously. Traffic switches with an atomic DNS-level swap — no restart, no dropped requests, users notice nothing.

BEFORE DEPLOY Production ──► 100% users ✅ [v1.2 running] Staging ──► internal only 🔶 [deploying v1.3] HEALTH CHECK PASSES → /health returns 200 SWAP (atomic, milliseconds) Production ──► 100% users ✅ [v1.3 now live] Staging ──► holds old v1.2 🔙 [instant rollback]
Rollback time: < 10 seconds. Old version stays warm in staging — one command swaps back if anything goes wrong in production.

Why this matters

  • Users experience 0 seconds of downtime per deploy
  • Staging is real production environment — catches config issues before users see them
  • Can push multiple times per day safely

Terraform config

resource "azurerm_linux_web_app_slot" "staging" {
  name           = "staging"
  app_service_id = azurerm_linux_web_app.chatbot.id
  site_config {
    health_check_path = "/health"
  }
}

Autoscaling — Azure Monitor Rules

Azure Monitor watches CPU and HTTP queue. It automatically adds or removes App Service instances — no manual intervention ever needed.

TriggerThresholdWindowAction
CPU high> 70%5 min+ 1 instance
HTTP queue> 10 requests5 min+ 1 instance
CPU low< 30%10 min− 1 instance
Capacity limits: Min 1 instance, Max 3 instances. Prevents runaway scaling on student credits. 5-minute cooldown between scale events.
🔥 Autoscale fired during our load tests! Azure Monitor sent an email alert confirming an instance was added when CPU exceeded 70%. Real proof the system works.
# Scale OUT — CPU > 70% for 5 min
rule {
  metric_trigger {
    metric_name  = "CpuPercentage"
    time_window  = "PT5M"
    operator     = "GreaterThan"
    threshold    = 70
  }
  scale_action {
    direction = "Increase"
    value     = "1"
    cooldown  = "PT5M"
  }
}
# HTTP Queue > 10 → instant scale out
rule {
  metric_trigger {
    metric_name  = "HttpQueueLength"
    threshold    = 10
  }
  scale_action {
    direction = "Increase"
    value     = "1"
    cooldown  = "PT3M"
  }
}
# Scale IN — CPU < 30% for 10 min
rule {
  metric_trigger {
    metric_name  = "CpuPercentage"
    time_window  = "PT10M"
    operator     = "LessThan"
    threshold    = 30
  }
  scale_action {
    direction = "Decrease"
    value     = "1"
    cooldown  = "PT10M"
  }
}

Structured Logging with Correlation IDs

# Every request gets a unique ID
@app.middleware("http")
async def telemetry(request, call_next):
    req_id = request.headers.get(
        "X-Request-ID") or uuid4()[:8]
    _request_id.set(req_id)
    response = await call_next(request)
    response.headers["X-Request-ID"] = req_id
    return response

Every log line includes the ID — trace a user's full journey:

{"level":"INFO","request_id":"a4f2","msg":"POST /chat/stream → 200 (847ms)"}

Security — Zero Trust Architecture

🔐

Azure Key Vault

All secrets stored in Key Vault. App Service has a User-Assigned Managed Identity — Azure handles token exchange automatically. Zero credentials in code, env vars, or CI logs.

# Terraform creates Key Vault + grants identity access
resource "azurerm_key_vault_secret" "groq_api_key" {
  name         = "groq-api-key"
  value        = var.groq_api_key
  key_vault_id = azurerm_key_vault.main.id
}
✅ HTTPS Only
https_only = true in Terraform. All HTTP redirects to HTTPS. TLS 1.2+ enforced by Azure.
✅ CORS Policy
Configurable via ALLOWED_ORIGINS env var. Controls which frontend domains can call the API.
✅ No Secrets in Code
All secrets are env vars. .env in .gitignore. CI/CD uses GitHub Secrets — never committed to git.
✅ Managed Identity
App reads Key Vault secrets at runtime via identity token. No username/password anywhere in the system.
✅ Soft-delete
Key Vault has 7-day retention. Secrets recoverable if accidentally deleted.

↓ if API unreachable (timeout 8s)
2. In-memory cache (last successful fetch)
↓ if cache empty (cold start)

  • Voice transcription fail → explains in chat
  • Streaming error → partial response shown

Professor Will Ask — Be Ready

Q: How does autoscaling work?
A: Azure Monitor watches CPU and HTTP queue. When CPU >70% for 5 min, it adds one App Service instance (max 3). When CPU <30% for 10 min, it removes one. We got an autoscale email alert — proof it fired in production.
Q: How many instances/pods are running?
A: Azure App Service uses "instances" (not pods — that's Kubernetes). Currently min 1, max 3. Autoscale adds/removes instances. Can show in portal under "Scale Out" → "Run history".
Q: What happens if the app crashes?
A: Azure polls /health every 60s. If it fails for 2 minutes, it auto-restarts the container. Zero manual action needed. Also the CI/CD won't promote broken code — staging health check blocks it.
Q: How is data kept fresh?
A: Inventory fetched from Mahesh's live API at startup, cached for 24h, refreshed by background task. Manual refresh via POST /inventory/refresh for demo. If API down, falls back to seed data (circuit breaker pattern).
Q: How are secrets managed?
A: Azure Key Vault. App Service has a Managed Identity — Azure issues tokens automatically. No credentials in code, Git, or CI/CD logs. Key Vault has soft-delete for recovery.
Q: What is RAG and why not fine-tuning?
A: RAG fetches live inventory data at query time. Fine-tuning bakes data into the model — can't be updated without retraining (expensive, slow). RAG means the bot always knows the current inventory.
