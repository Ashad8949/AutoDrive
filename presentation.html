What is RAG?

Retrieval Augmented Generation — the LLM doesn't memorise facts. Instead, it fetches relevant data at query time and uses it as context. Think of it as giving the LLM an open-book exam.

Answer

Step 1 — Retrieve

User asks "SUVs under ₹20 lakh". TF-IDF scores all 12 cars against this query. Top matches: Creta, Seltos, Hector, Nexon EV.

Step 2 — Augment

Matched car documents (price, features, location, rating) are injected into the LLM's system prompt as structured context before it generates any response.

Step 3 — Generate

LLaMA 3.3-70B sees the context and generates a response grounded in real data — correct prices in ₹, actual features, real cities. Never hallucinates.

Why RAG — Not Fine-tuning or Prompting?

ApproachData FreshnessCostAccuracyVerdict
Pure Prompting
Paste all cars into prompt		 ✅ Real-time⚠️ High token cost❌ Poor (> 30 cars) Fails at scale
Fine-tuning
Train model on inventory		 ❌ Static❌ Very expensive✅ Good Stale data
Vector RAG
Embeddings + FAISS		 ✅ Refreshable⚠️ Embedding API cost✅ Excellent Overkill for 12 cars
TF-IDF RAG ← Used
Keyword retrieval		 ✅ Live API (24h cache)✅ Free (no API)✅ Excellent for inventory ✓ Perfect fit

Why TF-IDF is right here

  • Car inventory = structured, keyword-rich text (make, model, fuel, price)
  • TF-IDF excels at exact keyword matching — "diesel SUV Mumbai" works perfectly
  • Zero API cost — runs locally, no embedding quota needed
  • Sub-millisecond retrieval for ≤ 30 cars

Upgrade path (for 1000+ cars)

  • Switch to Azure AI Search with vector embeddings
  • One config change in config.py
  • Architecture already supports this — retriever is pluggable
  • text-embedding-3-large deployment already exists in Azure OpenAI

Containerisation — Docker

Multi-stage build keeps the production image small and secure — build tools stay in the builder stage, only the runtime lands in production.

# Stage 1 — Builder (has pip, gcc, etc.)
FROM python:3.12-slim AS builder
WORKDIR /app
COPY requirements.txt .
# --prefix keeps packages isolated
RUN pip install --prefix=/install -r requirements.txt

# Stage 2 — Production (minimal)
FROM python:3.12-slim
WORKDIR /app
# Copy only the installed packages
COPY --from=builder /install /usr/local
COPY src/ ./src/
ENV PYTHONPATH=/app
CMD ["uvicorn", "src.main:app",
     "--host", "0.0.0.0", "--port", "8002"]

GitHub Container Registry

Free for public packages. Image pushed as ghcr.io/ashad8949/autodrive-chatbot:latest. Azure App Service pulls directly — no Azure Container Registry cost.

Image Tagging Strategy

  • :latest — always the newest build
  • :<git-sha> — immutable, traceable to commit
  • Azure pulls by SHA in staging, latest in prod swap

Layer Caching

GitHub Actions caches Docker layers — only changed layers re-build. Average build time: 45 seconds.

Test Gate

Unit tests run with mocked LLM. If any test fails, pipeline stops — code never ships to staging.

Build Gate

Docker build fails if Dockerfile has errors. Image only pushed if build succeeds. Tagged with git SHA for traceability.

Staging Gate

Deploys to autodrive-chatbot-staging slot. Polls /health for 3 minutes. Blocks if unhealthy.

Auto-Rollback

After production slot swap, re-checks /health. If it fails, automatically swaps back. Zero manual intervention.

Blue-Green Deployment — Zero Downtime

Azure App Service deployment slots run two versions simultaneously. Traffic switches instantly with a DNS-level swap — no restart, no dropped requests.

BEFORE DEPLOY Production slot ──► users (100% traffic) [old image v1.2] ✅ live Staging slot ────► internal only [new image v1.3] 🔶 being deployed HEALTH CHECK PASSES curl /health → 200 OK ✅ AFTER SWAP (atomic, milliseconds) Production slot ──► users (100% traffic) [new image v1.3] ✅ live, zero downtime Staging slot ────► now holds old v1.2 [old image v1.2] 🔙 instant rollback ready

Why this matters

  • Users experience 0 seconds of downtime during every deploy
  • Old version kept warm — rollback in under 10 seconds
  • Staging is a real production environment — catches config issues

Terraform config

resource "azurerm_linux_web_app_slot" "staging" {
  name           = "staging"
  app_service_id = azurerm_linux_web_app.chatbot.id
  site_config {
    health_check_path = "/health"
  }
}

CLI swap command

az webapp deployment slot swap \
  --name autodrive-chatbot \
  --slot staging \
  --target-slot production

Autoscaling — Azure Monitor Rules

Azure Monitor autoscale watches CPU and HTTP queue depth. It adds or removes App Service instances automatically — no manual intervention.

TriggerThresholdWindowAction
CPU high> 70%5 min + 1 instance
HTTP queue> 10 req5 min + 1 instance
CPU low< 30%10 min − 1 instance
Capacity limits: Min 1 instance, Max 3 instances. Prevents runaway scaling on student credits. Cooldown 5 min between scale events. 
# Scale OUT rule — CPU > 70%
rule {
  metric_trigger {
    metric_name      = "CpuPercentage"
    time_window      = "PT5M"
    operator         = "GreaterThan"
    threshold        = 70
  }
  scale_action {
    direction = "Increase"
    value     = "1"
    cooldown  = "PT5M"
  }
}

# Scale IN rule — CPU < 30%
rule {
  metric_trigger {
    metric_name  = "CpuPercentage"
    time_window  = "PT10M"
    operator     = "LessThan"
    threshold    = 30
  }
  scale_action {
    direction = "Decrease"
    value     = "1"
    cooldown  = "PT10M"
  }
}
HTTP Queue trigger also scales out when requests pile up — handles sudden traffic spikes (like a live demo) faster than CPU can react.

Structured Logging with Correlation IDs

# Every request injects a unique ID
@app.middleware("http")
async def request_telemetry(request, call_next):
    req_id = request.headers.get(
        "X-Request-ID") or uuid4()[:8]
    _request_id.set(req_id)   # thread-local
    response = await call_next(request)
    response.headers["X-Request-ID"] = req_id
    return response

Every log line includes the request ID — trace a single user's journey end-to-end through all log entries:

 "msg":"POST /chat/stream → 200  (847ms)"}

Security — Zero Trust Architecture

🔐

Azure Key Vault

All secrets (GROQ_API_KEY, AZURE_OPENAI_KEY) stored in Key Vault. App Service reads them at runtime using Key Vault references — secret never appears in app settings, logs, or code.

# In app_settings — not the actual key!
GROQ_API_KEY = "@Microsoft.KeyVault(
  SecretUri=https://autodrive-kv-.../
  secrets/groq-api-key/)"
🤝

Managed Identity

App Service has a User-Assigned Managed Identity. Key Vault grants read access to this identity. No username/password — Azure handles the token exchange automatically.

✅ HTTPS Only

https_only = true in Terraform. All HTTP requests redirect to HTTPS. TLS 1.2+ enforced by Azure.

✅ CORS Policy

Configurable via ALLOWED_ORIGINS env var. Restricts which frontend domains can call the API.

✅ No Secrets in Code

All secrets are environment variables. .env is in .gitignore. CI/CD uses GitHub Secrets — never committed.

✅ Soft-delete on Key Vault

7-day retention if secrets are accidentally deleted. Prevents catastrophic data loss.

Resilience — Self-Healing System

Health Check Auto-Restart

Azure polls /health every 60 seconds. If the endpoint fails to respond for 2 minutes, Azure automatically restarts the container — no manual intervention.

  "provider": "groq"

Inventory Fallback Pattern

1. Live API (autodriveai.duckdns.org)
↓ if API unreachable
2. In-memory cache (last successful fetch)
↓ if cache empty (cold start)
3. seed_data.json (emergency fallback)

Graceful Error Handling

  • LLM timeout → returns user-friendly error message
  • Voice transcription fail → explains in chat
  • Streaming error → partial response shown, error appended
  • All errors logged with correlation ID for tracing
